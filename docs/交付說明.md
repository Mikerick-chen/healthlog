## 3. ⭐ 資訊增益計算過程與結果（§6.3，門檻怎麼選出來的）

**標籤基準**：種子資料三組設計分布（高/中/低）。
**公式**：熵 `H(S) = -Σ p·log2(p)`；資訊增益 `IG = H(parent) - Σ(|Sv|/|S|)·H(Sv)`。
**程式**：[`InformationGainService.java`](../src/main/java/com/healthlog/service/InformationGainService.java)（每次啟動都會跑一次並 log，亦可由 `GET /health-logs/analysis` 重跑）。

### 計算流程（嚴格依 §6.3 step 3：遞迴子集合）

1. **第一層 T1**：在 90 筆全體上對三特徵各取最佳切點。三者 IG 相同（皆 0.85，因皆能把最極端的「高」組切出），依 §6.2 結構取 **睡眠 sleep_hours** 為根，最佳門檻 **T1 = 5.2 hr**。
2. **遞迴子集合**：用 T1 把資料切成 `睡眠<5.2`（25 筆，全為高，熵=0）與 `睡眠≥5.2`（65 筆，中+低混合，熵>0）。取**較混合的右子集合**續算。
3. **第二層 T2**：在右子集合上算步數最佳切點 → **T2 = 6111 步**（剛好落在普通組 ≤5800 與低風險組 ≥6500 之間）。
4. **第三層 T3**：在右子集合上算心情最佳切點 → **T3 = 7 分**（落在普通組 ≤6 與低風險組 ≥7 之間）。

> **為何要遞迴子集合（除錯關鍵）**：第一版若三特徵都在「整份資料」取最佳切點，會得到 T2≈3351、T3≈4，三者都只把「高」組切出來，導致**中組全被併入低組**（實測分布 65 低 / 25 高 / 0 中）。改為「對第一層切出的子集合重算 T2/T3」後，才找到區分「中 vs 低」的門檻，分布回正為 25/40/25。此即 §6.3 step 3 的精神。

### 最終門檻

| 層級 | 特徵 | 門檻 | 資訊增益(全體) |
|-|-|-|-|
| T1 | sleep_hours | 5.2 hr | 0.85 |
| T2 | steps | 6111 步 | 子集合上最佳 |
| T3 | mood_score | 7 分 | 子集合上最佳 |

## 4. 決策樹邏輯的程式碼位置

- 核心：[`DecisionTreeService.java`](../src/main/java/com/healthlog/service/DecisionTreeService.java)
  - 🔒 **多層分支**結構：第一層睡眠 → 第二層步數 → 第三層心情（非單一 if、非加權平均）。
  - ⭐ 明確處理「**步數正常但心情差**」中間情況（§5.2 檢查點）→ 判為「中」。
  - 回傳 `decisionPath`（走過的每一步），供前端決策路徑視覺化（§10.1 炫項）。
- 門檻由 §3 資訊增益實測回填（`applyThresholds`）。
- 風險判定皆為自寫程式邏輯，**未呼叫任何付費 AI API**（§A.2 / §12）。

## 5. API 端點（§7，全部實作）

| Method | Path | 用途 |
|-|-|-|
| GET | `/health-logs` | 取得全部（可加 `?from=&to=` 區間、前端可再做風險篩選） |
| POST | `/health-logs` | 新增（即時算 risk_level 並寫回） |
| PUT | `/health-logs/{id}` | 修改（重算 risk_level） |
| DELETE | `/health-logs/{id}` | 刪除 |
| GET | `/health-logs/risk?id=` | 🔒 依決策樹計算風險，回傳 riskLevel + reasoning + decisionPath |
| GET | `/health-logs/analysis` | ⭐ 資訊增益分析（§10.4，可重跑） |
| GET | `/health-logs/export.csv` | 匯出 CSV（§10.1） |

錯誤格式 `{ "error": "訊息" }`，對應 400/404/500（[`GlobalExceptionHandler.java`](../src/main/java/com/healthlog/controller/GlobalExceptionHandler.java)）。

## 7. 已知限制

1. **JDK 版本**：以 `--release 21` 編譯（class 版本 65），於本機 JDK 25 執行；換機請確保 JDK ≥ 21。
2. **SQLite 並發**：連線池設為 1（避免鎖庫），單機展示足夠，不適合高並發。
3. **資訊增益標籤來源**：啟動時用「設計組別」當標籤導出門檻；`/analysis` 端點則改用 DB 現有 `risk_level` 當標籤重算（兩者用途不同，已於回傳 `labelSource` 標明）。
4. **門檻為二元切分**：每層採單一門檻的二分法（非多元切分），符合 §6.2 結構；如需更細可擴充。
5. 種子資料的「中組」少數邊界值可能因亂數落在門檻附近，但整體分布穩定為 25/40/25（固定種子可重現）。
